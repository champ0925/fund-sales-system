# 基金销售系统技术设计文档

## 一、需求分析

### 1.1 场景理解

本项目是一个面向基金销售场景的管理系统，主要服务于基金销售人员。核心场景包括：

- **产品管理**：展示基金产品货架，支持产品的增删改查
- **客户管理**：管理客户信息，记录客户持仓和跟进情况
- **数据看板**：通过可视化图表展示关键业务数据
- **AI 智能助手**：利用大语言模型实现自然语言查询和数据解读

### 1.2 已实现的功能

1. **产品管理模块**
   - 产品列表展示（支持分页）
   - 新增、编辑、删除产品
   - 批量删除功能

2. **客户管理模块**
   - 客户列表管理
   - 客户持仓查看
   - 跟进记录管理

3. **数据看板**
   - 产品类型分布统计
   - 产品状态统计
   - 客户数量统计

4. **AI 智能助手**
   - 自然语言 SQL 查询
   - 图表自动生成（饼图、柱状图、折线图）
   - 产品新增功能（自然语言描述）

### 1.3 功能选择的原因

- 选择前后端分离架构，便于维护和扩展
- 采用 AI 助手是因为传统 CRUD 系统查询效率低，用自然语言可以大幅提升用户体验
- 选择通义千问作为 LLM 是因为国内访问稳定，成本较低

### 1.4 因时间有限未完成的功能

- 客户批量导入/导出
- 权限管理和用户认证
- 操作日志审计
- 详细的数据统计分析
- 移动端适配

### 1.5 如果再给我2天会优先做什么

1. **完善 AI 助手的错误处理和提示优化** - 提升用户体验
2. **增加数据导出功能** - 实际业务需求
3. **完善单元测试** - 保证代码质量

## 二、技术选型

### 2.1 技术方案选择

| 层级 | 技术选型 | 选择原因 |
|------|----------|----------|
| 前端 | React 19 + TypeScript | 生态成熟，类型安全，团队熟悉 |
| 前端 UI | Ant Design 6.x | 组件丰富，适合后台管理系统 |
| 前端图表 | Recharts | React 原生支持，集成方便 |
| 后端 API | Express.js | 轻量级，上手快，适合中小项目 |
| AI Agent | FastAPI + LangChain | Python 生态完善，LLM 集成方便 |
| LLM | 通义千问 (Qwen) | 国内可用，API 稳定，成本适中 |
| 数据库 | MySQL | 传统关系型数据库，适合业务系统 |
| ORM | SQLAlchemy | Python 生态主流 ORM |

### 2.2 为什么选这些而非其他

- **React vs Vue**: 团队对 React 更熟悉，Ant Design 组件库生态更好
- **FastAPI vs Django**: FastAPI 性能更高，异步支持更好，API 设计更现代化
- **通义千问 vs OpenAI**: 国内访问稳定，无需翻墙，成本更低
- **MySQL vs MongoDB**: 业务系统以结构化数据为主，MySQL 更适合

## 三、AI 协作日志

### 3.1 AI 使用环节

本项目在以下环节使用了 AI：

1. **代码生成**：AI 辅助生成部分重复性代码
2. **SQL 生成**：AI 将自然语言转换为 SQL 查询
3. **意图识别**：AI 判断用户问题是否需要生成图表
4. **答案生成**：AI 将查询结果转换为自然语言回答

### 3.2 AI 协作案例

#### 案例一：SQL 生成 prompt

**Prompt**:
```
你是一个基金数据查询助手，根据用户问题生成可执行的 MySQL SQL。
数据库表：
1. products(产品表) id,product_name,product_type,latest_nav,establish_scale,product_status,create_time
2. customers(客户表) id,customer_name,phone,company,customer_status,create_time
3. customer_hold(持仓表) id,customer_id,product_id,hold_amount,buy_time
4. customer_follow(跟进表) id,customer_id,follow_way,follow_content,follow_time,next_plan

只返回SQL，不要解释，不要多余内容。
用户问题：帮我查一下运作中的债券型基金有哪些
SQL：
```

**AI 输出**:
```sql
SELECT product_name, latest_nav, establish_scale 
FROM products 
WHERE product_type = '债券型' AND product_status = '运作中'
```

**我的处理**:
- 完全采纳，测试通过
- 将其整合到 LangChain Chain 中，配合 SQL 执行器使用

---

#### 案例二：图表意图识别 prompt

**Prompt**:
```
你是一个基金数据助手，帮我分析用户问题是否需要生成图表。
用户问题：帮我看看各类产品的数量分布

根据以下规则判断：
1. 如果用户要求生成"饼图"、"占比图"、"分布图"、"环形图"，返回 JSON：{"need_chart": true, "chart_type": "pie", "group_by": "分组字段", "value_field": "数值字段", "title": "图表标题"}
2. 如果用户要求生成"柱状图"、"条形图"、"柱形图"，返回 JSON：{"need_chart": true, "chart_type": "bar", "group_by": "分组字段", "value_field": "数值字段", "title": "图表标题"}
...

返回：
```

**AI 输出**:
```json
{"need_chart": true, "chart_type": "pie", "group_by": "product_type", "value_field": "count", "title": "各类产品数量分布"}
```

**我的处理**:
- 完全采纳
- 根据 AI 返回的 chart_type 和 group_by 字段，动态生成对应的 SQL 查询
- 在前端使用 Recharts 渲染对应的图表类型

#### 案例三：前端组件代码生成（已放弃）

**Prompt**:
```
用 React + Ant Design 写一个产品管理的增删改查页面，需要支持分页、搜索、批量删除
```

**AI 输出**:
生成了完整的组件代码，包含表格、表单、弹窗等

**我的处理**:
- 放弃使用，因为已有类似的实现

### 3.3 AI 帮助最多的地方

1. **SQL 生成** - 极大减少手动编写复杂 SQL 的工作
2. **代码片段参考** - 提供思路和最佳实践
3. **错误排查** - 快速定位问题原因

### 3.4 AI 帮不上忙的地方

1. **整体架构设计** - 需要人工判断业务场景和技术选型
2. **前端页面样式调整** - AI 生成的样式往往需要大量手动调整
3. **业务逻辑理解** - 对特定业务规则的理解需要人工介入

## 四、自我复盘

### 4.1 当前实现中有哪些你已知但来不及修的问题？

1. **AI Agent SQL 注入风险**
   - `handle_add_product` 函数中直接拼接 SQL 字符串
   - 应改用参数化查询

2. **前端路由刷新问题**
   -刷新页面会 404，需要配置 Vite 路由 history fallback

3. **错误处理不完善**
   - AI 接口超时没有好的降级策略
   - 数据库连接失败没有重试机制

4. **缺乏测试**
   - 没有单元测试和集成测试
   - 无法保证代码质量

### 4.2 如果这是真实的企业项⽬，你会做哪些不同的决策？

1. **架构层面**
   - 引入微服务架构，将 AI 服务独立部署
   - 增加 API 网关，统一鉴权和限流
   - 使用 Redis 缓存热点数据

2. **安全层面**
   - 完善的用户认证和权限管理
   - SQL 注入防护使用参数化查询
   - API 接口签名验证
   - 敏感数据加密存储

3. **工程化**
   - 完善的 CI/CD 流水线
   - 引入 ESLint + Prettier 代码规范
   - 增加单元测试和 E2E 测试
   - 使用 Docker 容器化部署

4. **运维监控**
   - 日志集中收集（ELK/EFK）
   - 业务指标监控（Prometheus + Grafana）
   - 链路追踪（SkyWalking）
   - 告警通知机制

5. **数据层面**
   - 数据库读写分离
   - 定期备份策略
   - 数据迁移脚本版本管理
